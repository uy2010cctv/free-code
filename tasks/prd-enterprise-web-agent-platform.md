# PRD: Enterprise Web Agent Platform

## Introduction

构建一个实时运行在 Web 端的企业 Agent 工作台。用户通过与 Agent 聊天，即可创建、编辑、计算和生成符合企业场景的 `docx` 和 `xlsx` 文件，并可在同一工作台中接入多种企业数据源、调用热更新的业务功能模块、生成企业报表。

该产品的目标不是做一个通用聊天机器人，而是做一个“最懂业务的系统”，让企业运营、财务、销售等职能团队在几乎零技术门槛下完成原本需要 ERP、CRM、表格、报表工具和人工协作才能完成的工作。长期目标是替代传统金蝶 ERP/CRM 一类系统中的高频业务操作与数据处理流程。

第一阶段聚焦“企业通用 Agent 工作台”，并以文档/表格生产、企业数据接入、模块化扩展、报表生成作为核心能力。不包含多人实时协同编辑。

## Goals

- 让企业业务用户通过自然语言完成常见文档与表格工作，无需学习复杂系统操作。
- 支持 Agent 创建、编辑、计算并输出符合企业场景的 `docx` 和 `xlsx` 文件。
- 支持以插件化方式接入多种企业数据源，并允许热更新新增数据源与业务模块。
- 支持基于企业数据与模板快速生成可复用的业务报表。
- 将一次任务的主要完成路径收敛到“聊天输入 -> Agent 执行 -> 工作区预览/修正 -> 导出结果”。
- 为后续替代传统 ERP/CRM 高频流程打下统一 Agent 平台基础。

## User Stories

### US-001: 创建企业业务会话
**Description:** 作为企业业务用户，我希望进入 Web 工作台后立即开始一个业务会话，这样我可以围绕单个任务持续和 Agent 协作。

**Acceptance Criteria:**
- [ ] 用户打开系统后可以创建新会话
- [ ] 每个会话有唯一 ID、标题、创建时间、最近活动时间
- [ ] 会话可以保存消息历史与关联的工作区文件
- [ ] 刷新页面后可以恢复已有会话列表
- [ ] Typecheck/lint passes
- [ ] Verify in browser using dev-browser skill

### US-002: 通过聊天驱动 Agent 执行业务任务
**Description:** 作为企业业务用户，我希望通过聊天描述任务，这样我无需学习命令或脚本即可让 Agent 帮我完成工作。

**Acceptance Criteria:**
- [ ] 用户可以在输入框中提交自然语言任务
- [ ] 消息流中清晰显示用户消息、Agent 回复、工具调用和工具结果
- [ ] Agent 执行过程中前端显示流式状态，而不是长时间无反馈
- [ ] 用户可以取消当前正在执行的任务
- [ ] Typecheck/lint passes
- [ ] Verify in browser using dev-browser skill

### US-003: 创建和编辑企业 Word 文档
**Description:** 作为企业业务用户，我希望 Agent 能创建和编辑企业文档，这样我可以快速完成合同、方案、通知、制度、纪要等工作。

**Acceptance Criteria:**
- [ ] Agent 可以创建新的 `docx` 文件
- [ ] Agent 可以追加、替换、插入文档内容
- [ ] Agent 可以基于企业语境生成常见文档结构，如标题、正文、表格、落款
- [ ] 用户可以在工作区预览生成的 Word 文档
- [ ] Typecheck/lint passes
- [ ] Verify in browser using dev-browser skill

### US-004: 创建和编辑企业 Excel 表格
**Description:** 作为企业业务用户，我希望 Agent 能创建和编辑企业表格，这样我可以完成预算、台账、销售清单、结算单和经营分析表。

**Acceptance Criteria:**
- [ ] Agent 可以创建新的 `xlsx` 文件
- [ ] Agent 可以写入单元格、插入工作表、设置公式、合并单元格和调整样式
- [ ] Agent 可以对企业业务数据执行计算并写入结果
- [ ] 用户可以在工作区预览 Excel 内容
- [ ] Typecheck/lint passes
- [ ] Verify in browser using dev-browser skill

### US-005: 在工作区查看和修订 Agent 产物
**Description:** 作为企业业务用户，我希望在聊天之外还能查看和修订文件，这样我可以确认结果是否符合业务要求。

**Acceptance Criteria:**
- [ ] 界面提供工作区面板，显示文件列表、编辑区和预览区
- [ ] 文本类文件可直接编辑并保存
- [ ] `docx` 和 `xlsx` 文件可在预览模式中查看
- [ ] 工作区可以标识未保存变更
- [ ] Typecheck/lint passes
- [ ] Verify in browser using dev-browser skill

### US-006: 接入企业数据源
**Description:** 作为企业业务用户，我希望 Agent 能读取企业数据源，这样我可以基于真实业务数据完成文档、表格和报表任务。

**Acceptance Criteria:**
- [ ] 系统支持注册多个数据源连接器
- [ ] 数据源至少支持文件类、数据库类和外部系统类三种类型
- [ ] Agent 在执行任务时可以读取指定数据源的数据
- [ ] 数据源配置在界面中可查看其名称、状态和可用能力
- [ ] Typecheck/lint passes
- [ ] Verify in browser using dev-browser skill

### US-007: 热更新业务功能模块
**Description:** 作为平台管理员，我希望可以在不重新部署整个前端的情况下增加业务模块，这样我可以快速上线新行业功能和报表能力。

**Acceptance Criteria:**
- [ ] 系统支持注册模块化功能单元
- [ ] 每个模块至少包含名称、描述、输入约束、可调用能力和版本信息
- [ ] 新模块注册后，Agent 可以在后续任务中调用该模块
- [ ] 模块变更不会破坏现有会话数据结构
- [ ] Typecheck/lint passes
- [ ] Verify in browser using dev-browser skill

### US-008: 生成企业报表
**Description:** 作为运营或财务用户，我希望 Agent 能基于企业数据和模板生成业务报表，这样我可以快速交付日报、周报、月报和专题分析。

**Acceptance Criteria:**
- [ ] 用户可以通过聊天要求生成指定类型的报表
- [ ] Agent 可以读取数据源并生成结构化报表内容
- [ ] 报表结果至少支持导出为 `docx` 或 `xlsx`
- [ ] 报表中明确标识数据来源和生成时间
- [ ] Typecheck/lint passes
- [ ] Verify in browser using dev-browser skill

### US-009: 保证数据计算结果可核对
**Description:** 作为企业业务用户，我希望 Agent 给出的计算和报表结果可追溯，这样我能确认关键数据是正确的。

**Acceptance Criteria:**
- [ ] 涉及计算的结果必须保留公式、来源数据或计算说明中的至少一种
- [ ] 报表界面或导出文件中能看到关键字段来源
- [ ] 当数据源缺失或字段不匹配时，系统返回明确错误，而不是静默生成错误结果
- [ ] Typecheck/lint passes
- [ ] Verify in browser using dev-browser skill

## Functional Requirements

1. FR-1: 系统必须提供基于 Web 的聊天式 Agent 工作台。
2. FR-2: 系统必须支持创建、查看、删除和恢复业务会话。
3. FR-3: 系统必须在会话中流式展示用户消息、Agent 回复、工具调用和工具结果。
4. FR-4: 系统必须支持 Agent 创建和编辑 `docx` 文件。
5. FR-5: 系统必须支持 Agent 创建和编辑 `xlsx` 文件。
6. FR-6: 系统必须提供工作区面板，用于查看文件树、预览文件和编辑文本文件。
7. FR-7: 系统必须允许用户取消当前正在运行的 Agent 请求。
8. FR-8: 系统必须为每个会话分配独立工作区，用于隔离生成的业务文件。
9. FR-9: 系统必须支持注册多个企业数据源连接器。
10. FR-10: 系统必须让 Agent 在任务执行中访问经授权的数据源。
11. FR-11: 系统必须支持模块化业务能力注册，并允许新增模块在不重构主流程的情况下被 Agent 调用。
12. FR-12: 系统必须支持基于数据源与模板生成企业报表。
13. FR-13: 系统必须在报表或计算结果中保留最少一层可追溯信息，用于核对数据正确性。
14. FR-14: 系统必须在数据源异常、字段缺失、工具执行失败时向用户返回明确错误信息。
15. FR-15: 系统必须支持将生成结果导出为企业常用交付格式，至少包括 `docx` 和 `xlsx`。

## Non-Goals

- 不做多人实时协同编辑。
- 不在第一阶段实现完整 ERP/CRM 替代范围内的全部业务流程。
- 不在第一阶段实现复杂审批流、组织权限模型和细粒度审计体系。
- 不在第一阶段实现完全自动执行的企业自动化中台，仍以“用户发起，Agent 辅助完成”为主。
- 不在第一阶段覆盖所有企业系统的双向深度集成。

## Design Considerations

- 界面应保持“聊天驱动 + 工作区辅助”的双栏结构，让非技术用户以最小学习成本开始使用。
- 工作区需要突出“当前文件”“未保存状态”“预览结果”，减少用户对 Agent 输出的不确定感。
- 工具调用结果不应只面向开发者展示，要能让业务用户理解 Agent 当前做了什么。
- 对 `docx`、`xlsx` 这类企业高频文件，应优先提供预览体验，而不是仅显示原始内容。
- 数据源、模块、报表模板建议设计成可发现、可选择、可解释的实体，而不是隐藏在 prompt 里。

## Technical Considerations

- 现有 Web 端已经具备会话、消息流、工作区、`docx/xlsx` 工具和基础文件操作能力，可作为第一阶段基础。
- 数据源接入应抽象成统一连接器接口，避免为每个企业系统硬编码调用逻辑。
- 模块热更新需要稳定的模块清单协议，至少包含模块元数据、版本、输入 schema 和执行入口。
- 报表生成应区分“原始数据读取”“计算处理”“模板渲染”“导出交付”四个阶段，便于排错和追踪。
- 数据正确性要求意味着系统不能只输出最终文字，必须保留中间结构化依据。
- 长期要替代 ERP/CRM 高频流程，后续需要补充权限、审计、版本管理和连接器治理，但不属于本阶段交付范围。

## Success Metrics

- 新用户在无培训条件下可在 10 分钟内完成首次文档或表格任务。
- 至少 80% 的高频试点任务可以通过“聊天 + 预览修订”流程完成，而无需切换到传统系统。
- 报表生成任务的关键数据字段可追溯率达到 100%。
- 文档或表格首次生成后，用户手动修改次数明显低于传统人工从零开始制作流程。
- 新增一个企业模块或报表模板的上线时间显著低于传统业务系统发版流程。

## Open Questions

- 第一批优先接入的企业数据源具体是哪几类，排序如何确定。
- 第一阶段是否需要区分“普通业务用户”和“平台管理员”的操作界面。
- 报表模板是由平台预置、管理员维护，还是允许业务用户自行沉淀模板。
- 数据源的授权与凭证管理是否在本项目内完成，还是依赖外部网关或配置中心。
- “热更新模块”最终采用前端注册、服务端注册，还是两者混合的机制。
